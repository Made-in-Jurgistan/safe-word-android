About This Guide Beginner

Scope, prerequisites, reading paths, and the philosophy behind this edition

Automatic Speech Recognition (ASR), also called Speech-to-Text (STT), is the technology that converts spoken audio into written text. A production-grade STT system is a multi-layer signal processing and machine learning pipeline that must handle acoustic noise, speaker variation, language modeling, streaming latency, battery constraints, and privacy requirements — simultaneously. This guide is about building that pipeline, not calling an API.

The difference between AssemblyAI.transcribe(file) and understanding what happens inside is the difference between using speech recognition and engineering it. When you're building a keyboard with voice input, an offline dictation app, a real-time voice agent, or anything that must work without a network connection, you need the full picture.

This 2026 edition reflects the decisive industry shift toward on-device inference. Models like Moonshine v2, whisper.cpp, and Sherpa-ONNX have matured to the point where sub-200ms latency is achievable on a mid-range Android phone without cloud round-trips. Every section has been written and code-verified against this new reality.

Why "Just Call the API" Is Sometimes Wrong

Cloud APIs — Deepgram, AssemblyAI, Google Cloud Speech-to-Text — are legitimate engineering choices for many products. This guide exists for when they are not:

  • Real-time voice agents — Cloud round-trips add 200–800ms. On-device avoids this entirely.
  • Offline applications — Field workers, embedded devices, remote deployments with no network.
  • Privacy-sensitive deployments — Medical, legal, or personal audio that must never leave the device.
  • Keyboard / IME integration — System-level voice input can't depend on internet availability.
  • Cost at scale — A cloud STT API at $0.006/minute costs $360/month for a single user transcribing 8 hours/day.

Section 03

Core Concepts & Terminology Beginner

WER, RTF, CTC, RNN-T, MFCC, log-mel — demystified. The vocabulary you need to read papers, documentation, and benchmark cards without confusion.

Before diving into any STT implementation, you need the vocabulary. This section defines every metric, acronym, and architecture term you will encounter in research papers, model cards, and API documentation — from WER and RTF to CTC, RNN-T, and attention-based decoders.

Accuracy Metrics

Word Error Rate (WER) is the standard accuracy metric for STT. It measures the minimum edit distance (Levenshtein distance) between the model output (hypothesis) and the ground truth (reference), normalized by reference length: 

Formula
WER = (Substitutions + Deletions + Insertions) / Total Reference Words × 100%

Example:
Reference:   "the quick brown fox jumps"   (5 words)
Hypothesis:  "the quick fox jumps"          (4 words — "brown" deleted)
WER = 1 deletion / 5 words = 20%

Lower is better. A WER of 5–7% is excellent for clear conversational English. Human transcriptionists achieve ~5%. Context matters enormously: 10% on studio audio is poor; 10% on factory-floor noise is outstanding.

Character Error Rate (CER) is similar but operates on characters, not words. Better for logographic languages (Chinese, Japanese) where "word" boundaries are ambiguous.

Speed Metrics

Real-Time Factor (RTF) measures processing time relative to audio duration: 

Formula
RTF = Processing Time / Audio Duration

RTF < 0.1  → 10× faster than real-time (excellent for streaming)
RTF = 0.3  → target threshold for real-time voice agents
RTF = 1.0  → exactly real-time (marginal for live use)
RTF > 1.0  → slower than real-time (batch only)

RTFx is the inverse: RTFx = 1/RTF. Higher = faster. NVIDIA Parakeet TDT achieves RTFx > 2000 on an A100 GPU. Moonshine Tiny achieves RTFx ≈ 20–40 on a Pixel 7.

Architecture Patterns

CTC (Connectionist Temporal Classification) is a loss function and decoding strategy for sequence models. It enables training without explicit audio-to-phoneme alignment, outputting per-frame token probabilities that collapse blanks and repetitions. CTC models are fast for streaming because every frame can produce output immediately — there's no need to wait for a full sequence.

Encoder-Decoder (Seq2Seq) is the dominant architecture for high-accuracy offline models (Whisper, Moonshine, Canary). The encoder embeds audio features into a latent space; the decoder autoregressively generates tokens conditioned on the encoded representation and previously generated tokens. Excellent accuracy but inherently non-streaming — the decoder needs the full encoded context.

RNN-Transducer (RNN-T / Transducer) is used by NVIDIA Parakeet and Google's production ASR. Combines a streaming encoder, a prediction network (language model branch), and a joint network. True streaming — outputs tokens as they're generated without waiting for end-of-utterance. The best architecture for latency-critical production deployments.

Ergodic Streaming Encoder (Moonshine v2) replaces full attention with sliding-window self-attention, achieving bounded TTFT (Time to First Token) independent of utterance length. This eliminates the fundamental latency limitation of encoder-decoder models for streaming use cases. See §11 Moonshine Deep Dive.

Feature Representations

Log-Mel Spectrogram is the input format for all modern end-to-end models (Whisper, Moonshine, Canary). Process: Short-Time Fourier Transform (STFT) → map frequency bins to mel scale → take log → normalize. Whisper uses 80 mel bins over a 25ms window with 10ms stride, producing an 80×3000 tensor for 30 seconds of audio.

MFCC (Mel-Frequency Cepstral Coefficients) is the traditional representation used by Kaldi, Vosk, and older systems. More compact than spectrograms but loses some information that modern models benefit from. Still relevant for embedded deployments with very tight memory budgets.

Raw Waveform — Moonshine's encoder directly convolves raw 16kHz samples without explicit feature extraction, simplifying the pipeline and enabling learnable feature representations.

Audio Capture Layer Beginner

Getting low-latency, correctly formatted audio from the microphone. The foundation of every STT pipeline — get this wrong and nothing downstream saves you.

AudioRecord is Android's low-level microphone API and the correct choice for STT. Do not use MediaRecorder — it compresses to AAC/MP4, adds encoder latency, and doesn't give you raw PCM. Do not use MediaRecorder.AudioSource.VOICE_RECOGNITION unless you want Android's built-in AGC applied invisibly — use MIC and handle your own signal chain.

Full AudioRecord Implementation (Kotlin Coroutines)

Kotlinimport android.media.*
import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*
import java.nio.ByteBuffer
import java.nio.ByteOrder

object AudioCaptureConfig {
    const val SAMPLE_RATE   = 16_000    // Hz — mandatory for STT models
    const val CHANNEL_IN    = AudioFormat.CHANNEL_IN_MONO
    const val ENCODING      = AudioFormat.ENCODING_PCM_16BIT
    const val FRAME_SAMPLES = 480       // 30ms @ 16kHz — good for VAD
    const val BUFFER_FRAMES = 10        // 300ms ring buffer
}

class AudioCaptureManager(private val scope: CoroutineScope) {

    private var audioRecord: AudioRecord? = null
    private var captureJob: Job? = null

    // Emits Float32 frames (normalized [-1, 1]) on a background thread
    private val _frames = MutableSharedFlow<FloatArray>(
        extraBufferCapacity = AudioCaptureConfig.BUFFER_FRAMES
    )
    val frames: SharedFlow<FloatArray> = _frames

    fun start() {
        if (audioRecord != null) return  // Already running

        val minBuf = AudioRecord.getMinBufferSize(
            AudioCaptureConfig.SAMPLE_RATE,
            AudioCaptureConfig.CHANNEL_IN,
            AudioCaptureConfig.ENCODING
        )
        // Use at least 2× minBuf to avoid overruns
        val bufBytes = maxOf(minBuf, AudioCaptureConfig.FRAME_SAMPLES * 2 * 2)

        audioRecord = AudioRecord(
            MediaRecorder.AudioSource.MIC,  // Use MIC, not VOICE_RECOGNITION
            AudioCaptureConfig.SAMPLE_RATE,
            AudioCaptureConfig.CHANNEL_IN,
            AudioCaptureConfig.ENCODING,
            bufBytes
        ).also { rec ->
            check(rec.state == AudioRecord.STATE_INITIALIZED) {
                "AudioRecord init failed — check RECORD_AUDIO permission"
            }
            rec.startRecording()
        }

        captureJob = scope.launch(Dispatchers.IO) {
            val buf = ShortArray(AudioCaptureConfig.FRAME_SAMPLES)
            while (isActive) {
                val read = audioRecord!!.read(buf, 0, buf.size)
                if (read > 0) {
                    // Convert Int16 PCM → Float32 normalized to [-1, 1]
                    val floats = FloatArray(read) { buf[it] / 32768f }
                    _frames.tryEmit(floats)
                }
            }
        }
    }

    fun stop() {
        captureJob?.cancel()
        captureJob = null
        audioRecord?.stop()
        audioRecord?.release()
        audioRecord = null
    }
}
⚠️
Always call getMinBufferSize() first

The minimum buffer size varies by device. A Pixel 7 returns 640 bytes; a Samsung S24 Ultra may return 1280 bytes. Creating an AudioRecord with a buffer smaller than the device minimum causes silent data loss — the AudioRecord initializes but returns garbage. Always call AudioRecord.getMinBufferSize() and use maxOf(minBuf, yourDesiredSize).

Kotlinclass VoiceKeyboardService : InputMethodService() {

    private lateinit var captureManager: AudioCaptureManager
    private lateinit var sttEngine: STTEngine

        super.onCreate()
        captureManager = AudioCaptureManager(serviceScope)
        sttEngine = STTEngine(applicationContext)
    }

    // Start recording when user activates voice input
    fun startVoiceInput() {
        captureManager.start()
        serviceScope.launch {
            captureManager.frames.collect { frame ->
                sttEngine.feedAudio(frame)
            }
        }
    }

    fun stopVoiceInput() {
        captureManager.stop()
        // Flush remaining audio, commit text to currentInputConnection
        serviceScope.launch {
            val finalText = sttEngine.flush()
            currentInputConnection?.commitText(finalText, 1)
        }
    }

    override fun onDestroy() {
        serviceScope.cancel()
        sttEngine.release()
        super.onDestroy()
    }
}

Section 05

Pre-Processing & Noise Reduction Intermediate

Noise suppression, echo cancellation, and automatic gain control — the signal conditioning layer that determines model accuracy before a single feature is computed.

Pre-processing is invisible when done correctly. It's catastrophically visible when done wrong or omitted. Models trained on clean studio audio suffer 30–60% WER degradation in noisy environments without pre-processing. This section covers every DSP stage between raw PCM and model input.

The Pre-Processing Chain

The recommended ordering is critical — each stage assumes the previous one has been applied:

  1. AEC (Acoustic Echo Cancellation) — Remove speaker playback captured by the microphone. Must happen first because NS and VAD will treat echo as speech.
  2. Noise Suppression (NS) — Attenuate background noise. RNNoise or WebRTC NS. Apply before AGC to avoid amplifying residual noise.
  3. AGC (Automatic Gain Control) — Normalize amplitude across speakers and environments.
  4. VAD — Gate the model on confirmed speech segments (see §06).

Tool 1: WebRTC Audio Processing Module (APM)

WebRTC APM is the battle-tested C++ library behind Google Meet, Zoom, and every major real-time audio platform. It provides AEC, NS, and AGC in a single API. For Android, the preferred approach is to use it via the webrtc-android AAR or build it from source using the libwebrtc CMake build.

Tool 2: RNNoise

RNNoise (Mozilla) is a small (~100KB) neural noise suppressor trained on a diverse noise corpus. It processes 48kHz audio in 10ms frames, requiring a 16kHz→48kHz resample before processing and 48kHz→16kHz resample after. It excels at stationary and semi-stationary noise (HVAC, traffic, fan). For non-stationary noise (crowd chatter), Krisp outperforms it significantly.

Tool 3: Android NoiseSuppressor (API 16+)

Android provides a system-level NoiseSuppressor as an AudioEffect. It's hardware-dependent — quality varies dramatically across devices. Use it as a fallback when you can't include a third-party NS library:

RMS-Based AGC (Python — for server-side pipelines)

Pythonimport numpy as np

def normalize_rms(audio: np.ndarray, target_rms: float = 0.1) -> np.ndarray:
    """RMS-based amplitude normalization. Safe for silence."""
    rms = np.sqrt(np.mean(audio ** 2))
    if rms < 1e-8:  # Silence — don't amplify
        return audio
    gain = target_rms / rms
    return np.clip(audio * gain, -1.0, 1.0)

class AdaptiveAGC:
    """Streaming AGC with asymmetric attack/release time constants."""
    def __init__(self, target_rms=0.1, attack=0.01, release=0.001):
        self.target_rms = target_rms
        self.attack  = attack    # Fast gain reduction
        self.release = release   # Slow gain increase (prevents pumping)
        self.current_gain = 1.0

    def process(self, frame: np.ndarray) -> np.ndarray:
        rms = np.sqrt(np.mean(frame ** 2) + 1e-10)
        desired = self.target_rms / rms
        alpha = self.attack if desired < self.current_gain else self.release
        self.current_gain = alpha * desired + (1 - alpha) * self.current_gain
        self.current_gain = np.clip(self.current_gain, 0.01, 10.0)
        return np.clip(frame * self.current_gain, -1.0, 1.0)
💡
The canonical pre-processing order

AEC → Noise Suppression → AGC → VAD → Feature Extraction. Never swap NS and VAD — your VAD will trigger on noise artifacts. Never apply AGC before NS — you'll amplify the noise you're about to suppress. Never skip AEC when your app plays audio the microphone can hear.

Section 06

Voice Activity Detection (VAD) Intermediate

Detecting when the user is actually speaking. The most impactful latency and battery optimization in the pipeline — reduces model compute by 80–90% and eliminates Whisper's hallucination-on-silence problem.

Without VAD, your STT model processes continuous audio including every second of silence and background noise. This wastes compute, drains battery, and causes Whisper — and most other encoder-decoder models — to hallucinate text during silence. A properly tuned VAD is often worth more accuracy points than a bigger model.

Tool 4: Silero VAD v6 (Recommended)

Silero VAD is the most widely-used VAD in production voice pipelines as of 2026. Key facts: ~2MB ONNX model, <1ms per 30ms chunk on CPU, >6000 language support, MIT license, no API keys. Version 6 (August 2025) introduced an "ifless" ONNX variant for even faster mobile inference and v6.2 (December 2025) added tinygrad support. Silero is used natively inside Moonshine Voice and can be enabled with a flag in whisper.cpp.

Silero VAD — Python Streaming Integration

Pythonimport numpy as np
import onnxruntime as ort
from typing import Optional, Callable

class SileroVADv6:
    """
    Silero VAD v6 ONNX wrapper for streaming STT pipelines.
    Operates on 30ms frames @ 16kHz (480 samples per chunk).
    Downloads model: pip install silero-vad
    """
    CHUNK_SIZE = 480   # 30ms @ 16kHz — v6 fixed window size
    SR = 16_000

    def __init__(self, threshold: float = 0.5, min_speech_ms: int = 250,
                 min_silence_ms: int = 500):
        from silero_vad import load_silero_vad
        self.model = load_silero_vad(onnx=True)
        self.threshold = threshold
        self.min_speech_samples = int(min_speech_ms * self.SR / 1000)
        self.min_silence_samples = int(min_silence_ms * self.SR / 1000)
        self._reset()

    def _reset(self):
        self.speaking = False
        self.speech_buffer: list[np.ndarray] = []
        self.silence_samples = 0
        self.speech_samples = 0

    def process_chunk(self, chunk: np.ndarray,
                       on_speech_end: Callable[[np.ndarray], None]) -> Optional[float]:
        """
        Feed a 30ms chunk (480 float32 samples @ 16kHz).
        Calls on_speech_end(audio) when a complete utterance is detected.
        Returns current VAD probability (0–1).
        """
        assert len(chunk) == self.CHUNK_SIZE, f"Expected {self.CHUNK_SIZE} samples"
        prob = float(self.model(chunk, self.SR).item())

        if prob >= self.threshold:
            self.speech_buffer.append(chunk.copy())
            self.speech_samples += self.CHUNK_SIZE
            self.silence_samples = 0
            self.speaking = True
        elif self.speaking:
            self.speech_buffer.append(chunk.copy())  # Include trailing silence
            self.silence_samples += self.CHUNK_SIZE
            if self.silence_samples >= self.min_silence_samples:
                if self.speech_samples >= self.min_speech_samples:
                    utterance = np.concatenate(self.speech_buffer)
                    on_speech_end(utterance)
                self._reset()
        return prob

Silero VAD — Android Kotlin via ONNX Runtime

Kotlin// build.gradle.kts dependency:
// implementation("com.microsoft.onnxruntime:onnxruntime-android:1.18.0")
// Copy silero_vad.onnx to app/src/main/assets/

import ai.onnxruntime.*
import java.nio.FloatBuffer

class SileroVAD(context: Context) : AutoCloseable {

    private val session: OrtSession
    private val env = OrtEnvironment.getEnvironment()
    private var h = OnnxTensor.createTensor(env, Array(2) { Array(1) { FloatArray(64) } })
    private var c = OnnxTensor.createTensor(env, Array(2) { Array(1) { FloatArray(64) } })

    init {
        val bytes = context.assets.open("silero_vad.onnx").readBytes()
        val opts = OrtSession.SessionOptions().apply {
            setIntraOpNumThreads(1)  // VAD runs on 1 thread — keeps it fast
            addNnapi()                // Use NNAPI if available on device
        }
        session = env.createSession(bytes, opts)
    }

    /**
     * Process one 480-sample (30ms @ 16kHz) Float32 chunk.
     * Returns speech probability 0.0–1.0.
     * Silero VAD is an RNN — state tensors h, c must persist between chunks.
     */
    fun predict(chunk: FloatArray): Float {
        require(chunk.size == 480) { "VAD requires exactly 480 samples (30ms)" }

        val inputTensor = OnnxTensor.createTensor(
            env, FloatBuffer.wrap(chunk), longArrayOf(1, 480)
        )
        val srTensor = OnnxTensor.createTensor(
            env, LongBuffer.wrap(longArrayOf(16000)), longArrayOf(1)
        )

        val inputs = mapOf("input" to inputTensor, "sr" to srTensor,
                           "h" to h, "c" to c)
        val result = session.run(inputs)

        val prob = (result[0].value as Array<*>)[0] as FloatArray
        h = result[1] as OnnxTensor  // Update RNN state
        c = result[2] as OnnxTensor

        inputTensor.close(); srTensor.close()
        return prob[0]
    }

    fun resetState() {
        h = OnnxTensor.createTensor(env, Array(2) { Array(1) { FloatArray(64) } })
        c = OnnxTensor.createTensor(env, Array(2) { Array(1) { FloatArray(64) } })
    }

    override fun close() { session.close() }
}
⚠️
Silero VAD is stateful — never skip resetState()

Silero VAD is an RNN. Its hidden state tensors h and c must be preserved between chunks within a single audio stream. When a new recording session starts, call resetState(). Forgetting this produces systematically wrong probability scores on the second and subsequent utterances.

Streaming & Chunking Intermediate

How to buffer, chunk, and stream audio between capture and model inference — including ring buffers, overlap strategies, and transport protocols.

Streaming is where latency is won or lost. A model that achieves RTF 0.05 on a test file can feel sluggish in a voice agent if the chunking strategy forces it to buffer 10 seconds before starting inference. This section covers every pattern for efficiently moving audio from capture to model.

Chunk Size vs. Latency vs. Accuracy

Larger chunks give models more context and typically improve accuracy; smaller chunks reduce first-token latency. The right balance depends on your model architecture:

Architecture Min Useful Chunk Optimal Streaming Chunk Notes
Whisper (encoder-decoder) Full utterance or 30s 30s padded segments Fixed-window model — doesn't stream natively
Moonshine v2 (ergodic) 30ms 480–960 samples Streaming-native, bounded TTFT regardless of length
Parakeet TDT (transducer) 40ms 160–320ms chunks Best streaming architecture for low latency
Vosk / Kaldi (CTC) 10ms 100–200ms CTC output per frame, greedy decoding

Ring Buffer Pattern (Kotlin)

Kotlin/**
 * Thread-safe ring buffer for audio samples.
 * Producer: AudioRecord callback thread.
 * Consumer: STT inference thread.
 */
class AudioRingBuffer(private val capacity: Int) {
    private val buffer = FloatArray(capacity)
    private var writePos = 0
    private var readPos  = 0
    private var available = 0

    @Synchronized
    fun write(samples: FloatArray) {
        for (s in samples) {
            buffer[writePos % capacity] = s
            writePos++
            if (available < capacity) available++
            else readPos++  // Overwrite oldest when full
        }
    }

    @Synchronized
    fun read(n: Int): FloatArray? {
        if (available < n) return null
        val out = FloatArray(n) { buffer[(readPos + it) % capacity] }
        readPos += n; available -= n
        return out
    }

    @Synchronized
    fun availableSamples(): Int = available
}

Feature Extraction Intermediate

Converting raw PCM into the tensor representations that acoustic models consume — log-mel spectrograms, MFCC, and why Moonshine bypasses this layer entirely.

Feature extraction transforms time-domain audio samples into frequency-domain representations that expose the spectral patterns models use to recognize phonemes. For most modern models — Whisper, Canary, Parakeet — this means computing an 80-bin log-mel spectrogram. For Moonshine v2, this layer is learnable convolutional layers inside the encoder itself.

Log-Mel Spectrogram (Python — matches Whisper's exact spec)

Pythonimport numpy as np

def log_mel_spectrogram(audio: np.ndarray, sr: int = 16000,
                         n_mels: int = 80, n_fft: int = 400,
                         hop_length: int = 160) -> np.ndarray:
    """
    Compute log-mel spectrogram matching Whisper's preprocessing exactly.
    Input:  float32 audio, normalized [-1, 1], mono 16kHz
    Output: float32 tensor of shape (n_mels, T) where T = ceil(len/hop_length)
    """
    import librosa

    # Compute mel spectrogram (power=2 for energy spectrogram)
    mel = librosa.feature.melspectrogram(
        y=audio, sr=sr, n_fft=n_fft, hop_length=hop_length,
        n_mels=n_mels, fmin=0, fmax=sr // 2
    )

    # Convert to log scale — clamp at 1e-10 to avoid log(0)
    log_mel = np.log10(np.maximum(mel, 1e-10))

    # Normalize to [-1, 1] range: Whisper uses (log_mel - max/8) × 4/max
    log_mel = np.maximum(log_mel, log_mel.max() - 8.0)
    log_mel = (log_mel + 4.0) / 4.0

    return log_mel.astype(np.float32)

# For 30-second audio at 16kHz: output shape is (80, 3000)
# 3000 = 30s × 16000 / 160 (hop_length)

Whisper Architecture Intermediate

How OpenAI's Whisper works inside — encoder-decoder attention, the 30-second window limitation, multilingual training, and why it hallucinates during silence.

Whisper (2022, OpenAI) remains the reference model for multilingual offline transcription. Its 99-language coverage, MIT license, and high accuracy on clean audio make it the default choice when mobile latency isn't critical. Understanding its architecture explains both its strengths and its limitations.

Architecture: Transformer Encoder-Decoder

Whisper is a standard encoder-decoder Transformer architecture. The encoder is a convolutional front-end followed by Transformer attention blocks that produce a context vector from the 30-second log-mel spectrogram. The decoder is an autoregressive Transformer that generates tokens by attending to both the encoded audio and previously generated tokens.

The 30-Second Window

Whisper's encoder always processes exactly 30 seconds of audio (padded with silence if shorter). This design choice makes Whisper excellent for batch transcription — a single forward pass encodes 30 full seconds — but problematic for streaming. The practical consequence:

  • Whisper cannot output any text until it has 30 seconds of audio (or until you force a shorter segment)
  • TTFT (Time to First Token) grows linearly with utterance length in the encoder
  • Short utterances waste compute on padding (a 2-second sentence pads 28 seconds of silence)
  • During silence, the decoder hallucinates — it was trained on real audio and generates plausible text for random noise

Moonshine v1 & v2 Expert

The streaming-native model family purpose-built for edge and mobile. v1's encoder/decoder state caching and v2's ergodic sliding-window attention — the architecture that makes sub-200ms latency real.

Moonshine (Useful Sensors / Moonshine AI) was designed from scratch to address Whisper's fundamental streaming limitation. Rather than forcing edge use into a batch architecture, Moonshine's encoder was redesigned for variable-length streaming with per-chunk state caching. Moonshine v2 (February 2026) takes this further with sliding-window ergodic attention that achieves bounded TTFT independent of utterance length.

Moonshine v1: Flexible Windows + State Caching

Moonshine v1's key innovation is that it processes variable-length audio rather than fixed 30-second windows. The encoder and decoder states are cached between audio chunks, enabling the model to "remember" what it already processed without recomputing from scratch. This alone delivers up to 5× faster processing compared to Whisper on the same hardware, because most of the encoder forward pass is skipped for incremental audio.

Moonshine v2: Ergodic Streaming Encoder

Moonshine v2 replaces full self-attention (quadratic in sequence length) with sliding-window self-attention in a position-free encoder. Every attention computation is local to a fixed window size, making the complexity O(N × W) rather than O(N²), where W is the window size. The "ergodic" property means the model provably accesses all historical context through overlapping windows.

The practical result: TTFT is bounded and fixed regardless of how long the user has been speaking. A 5-second utterance and a 2-minute monologue both produce the first token in the same time. This is physically impossible with a full-attention encoder.

            val bytes = context.assets.open(assetName).readBytes()
            return env.createSession(bytes, OrtSession.SessionOptions().apply {
                setIntraOpNumThreads(4)
                addNnapi()  // Hardware acceleration on supported devices
            })
        }
        // Models from: github.com/moonshine-ai/moonshine/releases
        preprocessSession = loadSession("moonshine_preprocess.onnx")
        encoderSession    = loadSession("moonshine_encoder.onnx")
        decoderSession    = loadSession("moonshine_decoder.onnx")
        tokenizer = Tokenizer.fromAsset(context, "moonshine_tokenizer.json")
    }

    /**
     * Transcribe a complete utterance (float32 mono 16kHz array).
     * For streaming, feed chunks to a running encoder state.
     * Returns the final transcript string.
     */
    fun transcribe(audio: FloatArray): String {
        // 1. Preprocess: raw audio → feature representation
        val audioTensor = OnnxTensor.createTensor(
            env, FloatBuffer.wrap(audio), longArrayOf(1, audio.size.toLong())
        )
        val features = preprocessSession.run(mapOf("audio" to audioTensor))
        audioTensor.close()

        // 2. Encode: features → context vectors
        val context = encoderSession.run(mapOf("features" to features[0]))

        // 3. Decode: autoregressive token generation
        val tokens = mutableListOf(tokenizer.bosToken)
        repeat(256) { // Max decode steps
            val logits = decoderSession.run(mapOf(
                "context"     to context[0],
                "input_ids"   to OnnxTensor.createTensor(env, tokens.toIntArray())
            ))
            val nextToken = argmax(logits[0] as FloatArray)
            if (nextToken == tokenizer.eosToken) return@repeat
            tokens.add(nextToken)
        }
        features.close(); context.close()
        return tokenizer.decode(tokens.drop(1))  // Skip BOS token
    }

    private fun argmax(arr: FloatArray): Int =
        arr.indices.maxByOrNull { arr[it] } ?: 0

    override fun close() {
        preprocessSession.close(); encoderSession.close(); decoderSession.close()
    }

Streaming Decoders Expert

CTC greedy decoding, beam search, RNN-Transducer, and how to implement each for real-time token emission.

The decoder translates the acoustic model's per-frame token probability distribution into a sequence of words. For encoder-decoder models (Whisper, Moonshine), decoding is autoregressive — token by token. For CTC models (Vosk, Wav2Vec2), decoding is frame-level probability collapse. The decoder choice determines whether your system can stream partial results or must wait for a complete utterance.

CTC Greedy Decode (Python)

Pythonimport numpy as np
from typing import List

def ctc_greedy_decode(log_probs: np.ndarray, blank_id: int = 0,
                       vocabulary: List[str] = None) -> str:
    """
    CTC greedy (best-path) decoding.
    log_probs: (T, V) — T frames, V vocabulary size
    Returns decoded string.
    """
    # Take argmax at each frame
    pred = np.argmax(log_probs, axis=1)  # shape: (T,)

    # CTC collapse: remove consecutive duplicates, then blanks
    collapsed = []
    prev = -1
    for token in pred:
        if token != prev:
            if token != blank_id:
                collapsed.append(token)
            prev = token

    if vocabulary:
        return ''.join(vocabulary[t] for t in collapsed)
    return collapsed  # Return token IDs if no vocabulary given

RNN-Transducer: Streaming-Native Decoder Architecture

CTC forces an independence assumption — every frame is decoded independently. The RNN-Transducer (RNN-T) removes that constraint: a joint network combines encoder and predictor (LM) outputs at every step, allowing the model to emit tokens before the full utterance is complete. Modern models that use RNN-T (including Moonshine v2 under the hood) achieve lower latency than CTC at the cost of more complex inference logic.

Emitting Partial Results During Inference

For a responsive UI, emit interim text as soon as CTC beam tokens stabilise across consecutive frames. The prefix that has not changed in the last N frames can be displayed as a committed partial result. 

Python# Streaming partial-result pattern (CTC beam search)
from pyctcdecode import BeamSearchDecoderCTC
import numpy as np

def stream_ctc_partial(
    decoder: BeamSearchDecoderCTC,
    logit_chunks,           # generator of numpy [T_chunk, V] arrays
    stability_frames: int = 5,
):
    # pyctcdecode context manager accumulates chunk logits
    with decoder.context_manager() as ctx:
        prev_prefix = ""
        for chunk in logit_chunks:
            ctx.decode_beams_batch([chunk])
            partial = ctx.decode()         # best beam so far
            if partial != prev_prefix:
                yield partial, False         # interim update
                prev_prefix = partial
        yield ctx.decode(), True           # final confirmed result

Kotlinobject TranscriptPostProcessor {

    // Number word → digit substitution (EN)
    private val numberMap = mapOf(
        "zero" to "0", "one" to "1", "two" to "2",
        "three" to "3", "four" to "4", "five" to "5",
        "six" to "6", "seven" to "7", "eight" to "8", "nine" to "9"
    )

    fun process(raw: String): String {
        var text = raw.trim()
        // Capitalize first letter of each sentence
        text = text.split(Regex("(?<=[.!?])\\s+"))
                   .joinToString(" ") { it.replaceFirstChar { c -> c.uppercase() } }
        // Capitalize "I" pronoun
        text = text.replace(Regex("""(?<=\s|^)i(?=\s|$)"""), "I")
        // Substitute number words
        numberMap.forEach { (word, digit) ->
            text = text.replace(Regex("""\b$word\b""", RegexOption.IGNORE_CASE), digit)
        }
        // Ensure sentence ends with period if no punctuation
        if (text.isNotEmpty() && text.last() !in ".!?") text += "."
        return text
    }
}

Filler Word & Disfluency Removal (Pre-LLM)

Before sending text to an LLM or committing to the input field, a lightweight rule-based pass can strip filler words, hesitation markers, and verbal ticks. This runs without any model, at near-zero latency, and dramatically improves readability. Aawaaz, Phemy, and Wispr Flow all implement this layer. The key insight: fillers should be configurable per-user because some words are fillers in casual speech but meaningful in other contexts (e.g., "like" in "I like pizza" vs. "I was like going to the store"). 

Kotlin/**
 * Configurable filler word removal engine.
 * Runs before LLM or punctuation model — pure regex, <1ms latency.
 *
 * Key design decisions (learned from Aawaaz & Wispr Flow):
 * 1. Fillers are configurable — users can add/remove from settings
 * 2. Word-boundary matching prevents false positives ("like" in "I like dogs")
 * 3. Repeated fillers ("um um um") collapse to nothing, not one "um"
 * 4. Preserve sentence structure — don't create orphaned punctuation
 */
class FillerWordRemover(
    fillers: Set<String> = DEFAULT_FILLERS,
) {
    companion object {
        val DEFAULT_FILLERS = setOf(
            "um", "uh", "erm", "ah", "hmm",
            "you know", "I mean", "basically",
            "literally", "right", "so",
        )
    }

    // Sort longest-first to prevent partial matches
    private val sortedFillers = fillers.sortedByDescending { it.length }

    // Build a single regex: \b(you know|I mean|basically|um|uh)\b\s*[,]?\s*
    private val fillerPattern: Regex = sortedFillers
        .joinToString("|") { Regex.escape(it) }
        .let { pattern ->
            Regex("""\b($pattern)\b[\s,]*""", RegexOption.IGNORE_CASE)
        }

    fun clean(text: String): String =
        fillerPattern.replace(text, " ")
            .replace(Regex("""\s{2,}"""), " ")
            .trim()
}

Self-Correction & Backtrack Detection (Pre-LLM)

Users naturally restate and correct themselves while speaking: "Send it to John… actually, send it to Sarah." Wispr Flow's "Backtrack" feature automatically detects these corrections and keeps only the final version. Aawaaz identifies trigger phrases like "actually no", "I mean", "wait", and "scratch that". This can be implemented as a rule-based pre-LLM pass or as part of the LLM prompt system.

The rule-based approach handles explicit triggers with zero latency. The LLM approach can also detect implicit restatements where the user naturally restates without a trigger word — Wispr Flow claims to identify when "you naturally restate something" and keep only the corrected version. A production system should layer both: rule-based for explicit triggers (fast path), LLM for implicit corrections (quality path). 

Kotlin/**
 * Self-correction / backtrack detection engine.
 *
 * Detects explicit correction triggers and removes the preceding clause.
 * Trigger phrases (from Wispr Flow, Aawaaz, and tambourine-voice):
 *   "actually" / "actually no" / "scratch that" / "wait" / "I mean" / "no wait"
 *
 * Algorithm:
 * 1. Scan for trigger phrase
 * 2. Find the clause boundary before the trigger (sentence start, comma, or "and"/"but")
 * 3. Remove everything from clause boundary to end of trigger phrase
 * 4. Keep text after the trigger as the correction
 *
 * Example: "Send it to John, actually send it to Sarah"
 *        → "Send it to Sarah"
 */
object BacktrackDetector {

    private val TRIGGERS = listOf(
        "actually no",   // Aawaaz-style explicit negation
        "scratch that",  // Wispr Flow / Aawaaz / tambourine
        "no wait",       // Common spoken correction
        "actually",      // Wispr Flow primary trigger
        "wait",          // Aawaaz trigger
        "I mean",        // Natural restatement marker
    )

    // Clause boundaries that scope what gets deleted
    private val CLAUSE_BOUNDARY = Regex("""[.!?,;]\s*|\b(and|but|so|then)\b\s*""")

    fun process(text: String): String {
        var result = text
        for (trigger in TRIGGERS) {
            val idx = result.lowercase().indexOf(trigger.lowercase())
            if (idx < 0) continue

            // Find the clause boundary before the trigger
            val beforeTrigger = result.substring(0, idx)
            val lastBoundary = CLAUSE_BOUNDARY.findAll(beforeTrigger)
                .lastOrNull()?.range?.last?.plus(1) ?: 0

            // Keep everything before clause + everything after trigger
            val afterTrigger = result.substring(idx + trigger.length).trimStart(',', ' ')
            result = result.substring(0, lastBoundary).trimEnd() +
                if (afterTrigger.isNotEmpty()) " " + afterTrigger else ""
            break  // Process one correction per pass; caller can loop
        }
        return result.trim()
    }
}

Kotlin/**
 * Voice punctuation command parser.
 * Scans raw transcript for spoken punctuation commands and substitutes symbols.
 *
 * Design notes (learned from Wispr Flow's Smart Formatting):
 * - Commands are case-insensitive and matched on word boundaries
 * - "new line" / "new paragraph" produce actual line breaks, not literal text
 * - Numbered list detection: "one... two... three..." → "1. ... 2. ... 3. ..."
 * - Order matters: longer phrases matched first ("exclamation point" before "point")

object VoicePunctuationParser {

    private data class Command(
        val spoken: String,
        val replacement: String,
        val attachLeft: Boolean = true,   // No space before symbol
        val attachRight: Boolean = false,  // Space after symbol
    )

    private val commands = listOf(
        // Formatting (highest priority)
        Command("new paragraph", "\n\n", attachLeft = true, attachRight = true),
        Command("new line", "\n", attachLeft = true, attachRight = true),
        // Multi-word punctuation (match before single-word)
        Command("exclamation point", "!"),
        Command("exclamation mark", "!"),
        Command("question mark", "?"),
        Command("full stop", "."),
        Command("em dash", "—"),
        Command("open parenthesis", "(", attachLeft = false, attachRight = true),
        Command("close parenthesis", ")"),
        Command("open quote", "\u201C", attachLeft = false, attachRight = true),
        Command("close quote", "\u201D"),
        Command("open bracket", "[", attachLeft = false, attachRight = true),
        Command("close bracket", "]"),
        Command("at sign", "@", attachLeft = true, attachRight = true),
        Command("pound sign", "#", attachLeft = true, attachRight = true),
        Command("dollar sign", "$", attachLeft = false, attachRight = true),
        Command("percent sign", "%"),
        Command("forward slash", "/", attachLeft = true, attachRight = true),
        Command("degree sign", "°"),
        Command("copyright sign", "©"),
        Command("plus sign", "+", attachLeft = true, attachRight = true),
        Command("equals sign", "=", attachLeft = true, attachRight = true),
        Command("greater than", ">", attachLeft = true, attachRight = true),
        Command("less than", "<", attachLeft = true, attachRight = true),
        // Single-word commands
        Command("period", "."),
        Command("comma", ","),
        Command("colon", ":"),
        Command("semicolon", ";"),
        Command("dash", "-"),
        Command("hyphen", "-"),
        Command("ellipsis", "…"),
        Command("apostrophe", "'", attachLeft = true, attachRight = true),
        Command("asterisk", "*"),
        Command("ampersand", "&"),
        Command("hashtag", "#", attachLeft = true, attachRight = true),
        Command("percent", "%"),
        Command("slash", "/", attachLeft = true, attachRight = true),
        Command("backslash", "\\", attachLeft = true, attachRight = true),
        Command("pipe", "|", attachLeft = true, attachRight = true),
        Command("tilde", "~"),
        Command("caret", "^"),
        Command("trademark", "™"),
        Command("equals", "=", attachLeft = true, attachRight = true),
    )

    fun parse(text: String): String {
        var result = text
        for (cmd in commands) {
            val pattern = Regex(
                """\s*\b${Regex.escape(cmd.spoken)}\b\s*""",
                RegexOption.IGNORE_CASE
            )
            result = pattern.replace(result) {
                val pre = if (cmd.attachLeft) "" else " "
                val post = if (cmd.attachRight) "" else " "
                pre + cmd.replacement + post
            }
        }
        return result.replace(Regex("""\s{2,}"""), " ").trim()
    }
}

Complete Pre-LLM Processing Pipeline

In a production voice keyboard, these pre-LLM stages run in sequence on the raw ASR output before any LLM formatting pass. The pipeline order matters: fillers first, then self-correction, then punctuation commands, then capitalization. 

Kotlin/**
 * Complete pre-LLM text processing pipeline.
 * Order: fillers → backtrack → punctuation commands → capitalisation → ITN
 * Total latency budget: <5ms for typical sentence lengths.
 */
object PreLLMPipeline {

    private val fillerRemover = FillerWordRemover()
    private val backtrackDetector = BacktrackDetector
    private val punctuationParser = VoicePunctuationParser

    fun process(raw: String): String {
        var text = raw.trim()
        // Step 1: Remove filler words (um, uh, you know, basically...)
        text = fillerRemover.clean(text)
        // Step 2: Detect and apply self-corrections ("actually", "scratch that")
        text = backtrackDetector.process(text)
        // Step 3: Parse spoken punctuation commands ("period" → ".")
        text = punctuationParser.parse(text)
        // Step 4: Basic capitalization & cleanup
        text = TranscriptPostProcessor.process(text)
        return text
    }
}

Multi-Part Prompt Architecture (tambourine-voice Pattern)

The open-source tambourine-voice project (321★) pioneered a powerful three-part prompt system that separates concerns for dictation LLM processing. This architecture is more maintainable and configurable than a single monolithic prompt. Aawaaz and Phemy adopted variations of the same pattern.

Prompt Part Responsibility When It Runs
Main Prompt Filler removal, punctuation, capitalization, sentence combining Every transcription
Advanced Prompt Backtrack corrections, list formatting, implicit restatement detection Every transcription (appended to Main)
Dictionary Prompt Custom word mappings, technical terms, proper nouns, user corrections When personal dictionary is non-empty 
Kotlin/**
 * Multi-part prompt builder for LLM formatting.
 * Inspired by tambourine-voice's 3-part system, adapted for on-device Qwen/Gemma.
 *
 * Why 3 parts instead of 1:
 * - Main prompt is stable and rarely changes
 * - Advanced prompt can be toggled (backtrack on/off)
 * - Dictionary prompt is per-user and dynamically generated
 * - Smaller, focused prompts reduce LLM confusion and hallucination
 */
class DictationPromptBuilder(
    private val personalDictionary: Map<String, String> = emptyMap(),
    private val enableBacktrack: Boolean = true,
) {
    // ─── Part 1: Main Prompt (always active) ───
    private val mainPrompt = """
You are a dictation cleanup assistant. Clean raw speech-to-text output:
1. Add proper punctuation (periods, commas, question marks)
2. Fix capitalization (sentences, proper nouns)
3. Remove filler words (um, uh, like, you know, basically)
4. Combine run-on fragments into proper sentences
5. Preserve the speaker's exact meaning — never add or change content
Return ONLY the cleaned text, nothing else.
"""

    // ─── Part 2: Advanced Prompt (backtrack + formatting) ───
    private val advancedPrompt = """
Additional rules:
- If the user corrects themselves (says "actually", "I mean", "scratch that",
  "wait", or naturally restates something), keep ONLY the corrected version.
  Remove the original phrasing and any correction markers.
- If the user dictates a numbered list ("one... two... three..."), format it
  as a proper numbered list with line breaks.
- If the user says "new line" or "new paragraph", insert the appropriate
  line break — do not write the words literally.
"""

    // ─── Part 3: Dictionary Prompt (per-user, dynamic) ───
    private fun buildDictionaryPrompt(): String {
        if (personalDictionary.isEmpty()) return ""
        val entries = personalDictionary.entries.joinToString("\n") { (from, to) ->
            "- \"$from\" → \"$to\""
        }
        return """
Personal dictionary — always apply these corrections:
$entries
"""
    }

    fun buildPrompt(rawTranscript: String): String = buildString {
        append(mainPrompt)
        if (enableBacktrack) append(advancedPrompt)
        append(buildDictionaryPrompt())
        append("\n\nRaw transcript:\n$rawTranscript\n\nCleaned:")
    }
}

Complete Wired Pipeline

Kotlin/**
 * Complete STT pipeline wired with Kotlin Flow.
 * Capture → VAD → Buffer → Whisper/Moonshine → Post-process → UI
 */
class STTPipeline(
    private val context: Context,
    private val onPartialTranscript: (String) -> Unit,
    private val onFinalTranscript: (String) -> Unit
) {
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main)
    private val capture = AudioCaptureManager(scope)
    private val vad = SileroVAD(context)
    private val stt = MoonshineSTT(context)  // or WhisperLib wrapper
    private val postproc = TranscriptPostProcessor

    // VAD state machine
    private val speechBuffer = mutableListOf<FloatArray>()
    private var isSpeaking = false
    private var silenceFrames = 0
    private val SILENCE_THRESHOLD_FRAMES = 17  // 500ms of silence @ 30ms/frame

    fun start() {
        capture.start()
        scope.launch(Dispatchers.IO) {
            capture.frames.collect { frame ->
                val speechProb = vad.predict(frame)

                when {
                    speechProb > 0.5f -> {
                        speechBuffer.add(frame)
                        isSpeaking = true
                        silenceFrames = 0
                    }
                    isSpeaking -> {
                        speechBuffer.add(frame)
                        silenceFrames++
                        if (silenceFrames >= SILENCE_THRESHOLD_FRAMES) {
                            processUtterance()
                        }
                    }
                }
            }
        }
    }

    private suspend fun processUtterance() {
        if (speechBuffer.size < 8) {  // Skip utterances < 240ms
            speechBuffer.clear(); isSpeaking = false; silenceFrames = 0
            return
        }
        val audio = speechBuffer.flatMap { it.toList() }.toFloatArray()
        speechBuffer.clear(); isSpeaking = false; silenceFrames = 0; vad.resetState()

        val rawText = stt.transcribe(audio)
        val cleanText = postproc.process(rawText)
        withContext(Dispatchers.Main) { onFinalTranscript(cleanText) }
    }

    fun stop() { capture.stop(); scope.cancel() }
}
⚠️

If you call transcribe() on the UI thread, Android will trigger an ANR after 5 seconds of accumulated blocking. Always dispatch inference to Dispatchers.Default or a dedicated newSingleThreadContext("stt").

import Foundation
import onnxruntime_objc

class MoonshineONNX {
    private let env: ORTEnvironment
    private let encoder: ORTSession
    private let decoder: ORTSession

    init() throws {
        env = try ORTEnvironment(loggingLevel: .warning)
        let opts = try ORTSessionOptions()
        try opts.setIntraOpNumThreads(2)
        let bundle = Bundle.main
        encoder = try ORTSession(
            env: env,
            modelPath: bundle.path(forResource: "moonshine_tiny_encoder", ofType: "onnx")!,
            sessionOptions: opts)
        decoder = try ORTSession(
            env: env,
            modelPath: bundle.path(forResource: "moonshine_tiny_decoder", ofType: "onnx")!,
            sessionOptions: opts)
    }

    func transcribe(pcm: [Float]) throws -> [Int64] {
        let pcmData = pcm.withUnsafeBytes { Data($0) }
        let pcmTensor = try ORTValue(
            tensorData: NSMutableData(data: pcmData),
            elementType: .float,
            shape: [1, NSNumber(value: pcm.count)])

        let encOut = try encoder.run(
            withInputs: ["pcm": pcmTensor],
            outputNames: ["last_hidden_state"],
            runOptions: nil)
        let hidden = encOut["last_hidden_state"]!

        var tokenIds: [Int64] = [1]    // BOS token
        for _ in 0..<200 {
            let tokData = tokenIds.withUnsafeBytes { Data($0) }
            let tokTensor = try ORTValue(
                tensorData: NSMutableData(data: tokData),
                elementType: .int64,
                shape: [1, NSNumber(value: tokenIds.count)])

            let decOut = try decoder.run(
                withInputs: ["encoder_hidden_states": hidden,
                             "input_ids": tokTensor],
                outputNames: ["logits"],
                runOptions: nil)

            let logits = try decOut["logits"]!.tensorData() as Data
            let floats = logits.withUnsafeBytes {
                Array(UnsafeBufferPointer<Float>(
                    start: $0.bindMemory(to: Float.self).baseAddress!,
                    count: $0.count / 4))
            }
            let vocabSize = floats.count / tokenIds.count
            let lastLogits = Array(floats.suffix(vocabSize))
            let nextToken = Int64(lastLogits.indices.max {
                lastLogits[$0] < lastLogits[$1]
            } ?? 2)
            if nextToken == 2 { break }    // EOS
            tokenIds.append(nextToken)
        }
        return tokenIds
    }
}

Kotlinclass StreamingSTTClient(
    private val serverUrl: String,
    private val onTranscript: (String, Boolean) -> Unit,
) {
    private val client = OkHttpClient.Builder()
        .readTimeout(0, TimeUnit.MILLISECONDS)   // No timeout for streaming
        .build()

    private var webSocket: WebSocket? = null

    fun connect(authToken: String) {
        val request = Request.Builder()
            .url(serverUrl)
            .header("Authorization", "Bearer $authToken")
            .build()
        webSocket = client.newWebSocket(request, object : WebSocketListener() {
            override fun onMessage(ws: WebSocket, text: String) {
                val json = Json.decodeFromString<TranscriptChunk>(text)
                onTranscript(json.text, json.isFinal)
            }
        })
    }

    /** Send a 20 ms PCM-16 chunk (640 bytes at 16 kHz mono) */
    fun sendAudio(pcm16: ByteArray) {
        webSocket?.send(ByteString.of(*pcm16))
    }

    /** Signal end-of-utterance so the server flushes remaining buffers */
    fun endOfSpeech() { webSocket?.send("EOS") }

    fun close() { webSocket?.close(1000, "done") }
}

Streaming with keep_ms

For low-latency voice input, use chunked processing: feed 1–3 s windows and keep the last 200 ms of each window as context for the next call. This avoids splitting words at chunk boundaries without running a full encoder pass on the full utterance. 

C// windowed streaming — keep 200 ms of audio at start of each new chunk
#define SAMPLE_RATE    16000
#define KEEP_MS        200
#define STRIDE_MS      1500        // process 1.5 s per call

static float s_keep[SAMPLE_RATE * KEEP_MS / 1000];  // 3200 samples
static int   s_keep_n = 0;

void process_chunk(struct whisper_context *ctx, const float *new_pcm, int n) {
    // Prepend previous-window tail for continuity
    float window[SAMPLE_RATE * (KEEP_MS + STRIDE_MS) / 1000];
    memcpy(window,           s_keep,    s_keep_n * sizeof(float));
    memcpy(window + s_keep_n, new_pcm,  n      * sizeof(float));
    int total = s_keep_n + n;

    // Save last KEEP_MS samples as context for next chunk
    int keep = min(SAMPLE_RATE * KEEP_MS / 1000, total);
    memmove(s_keep, window + total - keep, keep * sizeof(float));
    s_keep_n = keep;

    struct whisper_full_params p = whisper_full_default_params(WHISPER_SAMPLING_GREEDY);
    p.n_threads = 4;  p.single_segment = true;  p.no_timestamps = true;
    whisper_full(ctx, p, window, total);

    const char *text = whisper_full_get_segment_text(ctx, 0);
    if (text) on_partial_result(text);
}

Kotlin (build.gradle)implementation("com.microsoft.onnxruntime:onnxruntime-android:1.19.0")
Kotlinclass MoonshineAndroid(private val assets: AssetManager) {

    private val env = OrtEnvironment.getEnvironment()
    private val opts = OrtSession.SessionOptions().apply {
        setIntraOpNumThreads(2)
        setInterOpNumThreads(1)
        addNNAPI()              // Delegate to Android Neural Networks API
    }

    private val encoder = env.createSession(
        assets.open("moonshine/encode.onnx").readBytes(), opts)
    private val decoder = env.createSession(
        assets.open("moonshine/uncached_decode.onnx").readBytes(), opts)

    fun transcribe(pcm16k: FloatArray): String {
        // --- Encode ---
        val pcmTensor = OnnxTensor.createTensor(
            env, FloatBuffer.wrap(pcm16k),
            longArrayOf(1, pcm16k.size.toLong()))
        val encResult = encoder.run(mapOf("pcm" to pcmTensor))
        val hidden    = encResult[0] as OnnxTensor

        // --- Greedy decode ---
        val tokenIds = mutableListOf(1L)   // BOS
        repeat(200) {
            val tokArr    = tokenIds.toLongArray()
            val tokTensor = OnnxTensor.createTensor(
                env, LongBuffer.wrap(tokArr),
                longArrayOf(1, tokArr.size.toLong()))
            val decOut = decoder.run(
                mapOf("encoder_hidden_states" to hidden,
                      "input_ids" to tokTensor))
            val logits = (decOut[0] as OnnxTensor).floatBuffer
            val vocabSize = logits.capacity() / tokArr.size
            val offset    = (tokArr.size - 1) * vocabSize
            var best = 0L; var bestVal = Float.NEGATIVE_INFINITY
            for (i in 0 until vocabSize) {
                val v = logits.get(offset + i)
                if (v > bestVal) { bestVal = v; best = i.toLong() }
            }
            if (best == 2L) return@repeat    // EOS
            tokenIds.add(best)
        }
        return tokenizer.decode(tokenIds)
    }
}

  • Stop recording immediately on focus loss — In an IME, call captureManager.stop() in onFinishInput(). In a standard app, stop in onPause().
  • Implement push-to-talk rather than always-on — Require an explicit gesture to activate the microphone for any voice feature that isn't a dedicated voice assistant.

GDPR / CCPA Considerations

  • If you use cloud STT for any users in the EU/EEA, you must have a Data Processing Agreement with the provider and declare audio processing in your privacy policy.
  • On-device processing with no transmission is categorically exempt from many cloud data processing obligations, but you must still disclose microphone access in your privacy policy.

⚠️
Audio data breaches are catastrophic. Unlike leaked passwords, leaked voice recordings cannot be changed. A single exposure can compromise biometric voiceprints, reveal medical conversations, and create legal liability under GDPR Article 9 (special category data). Design your architecture to ensure raw audio never touches persistent storage or network I/O unless explicitly required by the product.

Encryption & Data-at-Rest

Even on-device architectures must protect intermediate artifacts. Follow these guardrails:

  • Models — Store ONNX/GGML model files in the app's internal storage (context.filesDir), which is encrypted at rest via Android file-based encryption (FBE) on all devices running Android 10+.
  • Transcripts — If you persist transcripts locally (e.g., for user history), use EncryptedSharedPreferences or EncryptedFile from the Jetpack Security library.
  • Language model priors — Personalisation data (n-gram frequencies, custom vocabulary) should be stored in the app sandbox and excluded from cloud backup via android:allowBackup="false" or backup rules.
  • Debug logs — Never log audio samples, transcript fragments, or model outputs at any log level in release builds. Use a BuildConfig.DEBUG guard on all verbose logging.

    if avg_logprob < -1.0:  return True        # Low confidence output

            if e.code == 1008:

Backtrack & Self-Correction Engine Expert

Deep dive into detecting and resolving user self-corrections during dictation. Covers trigger-word detection, clause-scoped deletion, implicit restatement via LLM, and the layered architecture used by Wispr Flow, aawaaz, and tambourine-voice.

Humans don't dictate perfectly. They say "the meeting is at three pm actually four pm" and expect the system to output only "the meeting is at 4 PM." Backtrack detection is what separates a transcription tool from a dictation tool.

The Backtrack Problem

Self-correction in speech comes in two forms — explicit (triggered by marker phrases) and implicit (natural restatement without any trigger). A production-grade dictation system must handle both.

Type Example Expected Output Detection Method
Explicit — single trigger "send it on Monday actually Tuesday" "send it on Tuesday" Keyword matching
Explicit — phrase trigger "I need the report scratch that I need the presentation" "I need the presentation" Keyword matching
Explicit — multiple corrections "set alarm for 6 no wait 7 actually 7:30" "set alarm for 7:30" Iterative keyword scan
Implicit — semantic restatement "the project costs about fifty thousand, it's around fifty K" "the project costs around $50K" LLM semantic analysis
Implicit — mid-sentence restart "we should probably, let's go ahead and approve it" "let's go ahead and approve it" LLM disfluency detection

Trigger Word Vocabulary

Across Wispr Flow, aawaaz, and tambourine-voice, the following trigger words are consistently recognised. Sorted by frequency-of-use from real dictation data:

Trigger Scope Behaviour
actually Previous clause or last noun/value Replace preceding value with following correction
I mean Previous clause Replace preceding clause
scratch that Previous sentence / previous dictation segment Delete everything since last sentence boundary
no wait / wait Previous clause Replace preceding clause
actually no Previous sentence Delete entire previous sentence
never mind Entire pending buffer Discard all uncommitted text
or rather Previous phrase Replace preceding phrase
correction Previous word or value Replace last word with following word

Layered Backtrack Architecture

The most effective approach — used by Wispr Flow — is a two-layer system where a fast rule-based engine handles explicit triggers (sub-millisecond) and an LLM handles implicit restatement (50–400ms). This avoids LLM latency for the common case while still catching the hard cases. 

Kotlin/**
 * Two-layer backtrack engine.
 *
 * Layer 1: Rule-based trigger detection (fast, deterministic)
 *   - Scans for known trigger words
 *   - Deletes preceding clause/sentence/value
 *   - Runs in <1ms — no LLM needed
 *
 * Layer 2: LLM implicit restatement (slow, semantic)
 *   - Asks the LLM to detect and resolve implicit self-corrections
 *   - Only runs when Layer 1 finds no triggers AND confidence is low
 *   - 50–400ms latency depending on model size
 */
class LayeredBacktrackEngine(
    private val ruleBasedDetector: BacktrackDetector,
    private val llmFormatter: Qwen3DictationFormatter?,
) {
    data class BacktrackResult(
        val text: String,
        val backtrackDetected: Boolean,
        val layer: Int,  // 1 = rule-based, 2 = LLM
    )

    suspend fun process(
        rawTranscript: String,
        foregroundPackage: String? = null,
    ): BacktrackResult {
        // Layer 1: fast rule-based scan
        val layer1Result = ruleBasedDetector.resolveBacktracks(rawTranscript)
        if (layer1Result != rawTranscript) {
            return BacktrackResult(layer1Result, backtrackDetected = true, layer = 1)
        }

        // Layer 2: LLM implicit detection (only if LLM is available)
        if (llmFormatter != null) {
            val llmResult = llmFormatter.format(rawTranscript, foregroundPackage)
            val changed = llmResult.length < rawTranscript.length * 0.85
            return BacktrackResult(llmResult, backtrackDetected = changed, layer = 2)
        }

        return BacktrackResult(rawTranscript, backtrackDetected = false, layer = 0)
    }
}

Clause Boundary Detection

When a trigger like "actually" is detected, the engine must determine how much text to delete. The answer depends on clause boundaries — punctuation, conjunctions, and natural pause points. The regex approach used by aawaaz and adapted in Section 14's BacktrackDetector: 

Regex// Clause boundary pattern — splits text at natural correction points
(?:^|[.!?]\\s+|,\\s+|\\s+(?:and|but|or|so|then|because)\\s+)

// Example: "send the email to John, actually to Sarah"
//   Clauses: ["send the email to John", "actually to Sarah"]
//   Trigger "actually" found in clause[1]
//   Delete clause[0]'s last value → replace with clause[1]'s value
//   Result: "send the email to Sarah"
⚠️
Deletion scope matters. "Scratch that" should delete the previous sentence, not the previous word. "Actually" should replace the previous value, not the previous sentence. Getting scope wrong is worse than not detecting the backtrack at all — users lose content they intended to keep.

Wispr Flow's Approach

Wispr Flow's proprietary backtrack engine is their most distinctive feature. Based on public documentation and user reports:

  • Explicit triggers — "actually" and "scratch that" are confirmed
  • Implicit detection — Wispr detects natural restatement without any trigger word, reportedly using a fine-tuned LLM
  • Scope intelligence — "actually" replaces only the preceding value; "scratch that" deletes the entire previous dictation segment
  • No false positives — "I actually think this is good" does NOT trigger deletion because "actually" is used as an adverb, not a correction marker
ℹ️
The false-positive problem. Naive keyword matching on "actually" fires on sentences like "I actually agree with you." Wispr Flow reportedly avoids this by checking whether the word following "actually" is semantically a replacement for what precedes it. This requires either (a) a POS-tagger to detect adverb vs conjunction usage, or (b) an LLM prompt that explicitly handles both cases.

Personal Dictionary & Auto-Learning Intermediate

Architecture for user-managed and auto-learned personal dictionaries that improve recognition of names, jargon, and domain-specific vocabulary. Covers manual entries, contacts import, CSV bulk import, auto-learning from corrections, and fuzzy matching for voice shortcuts.

Every user has words that no general ASR model knows — their colleagues' names, product names, technical acronyms, and slang. A personal dictionary bridges this gap by post-processing ASR output to replace misrecognised words with the user's preferred spellings.

Dictionary Entry Types

Type Source Example Priority
Manual entry User types in settings "kubernetes" → "Kubernetes" Highest — user explicitly defined
Contacts import Device contacts provider "john smith" → "John Smith" High — real names with known spelling
CSV bulk import User-uploaded file Medical/legal terminology lists Medium — batch import, less trusted
Auto-learned User corrections over time "kubernetes" → "Kubernetes" (after 3 corrections) Low — requires confirmation threshold

Dictionary Architecture

Kotlin/**
 * Personal dictionary with priority-based replacement.
 * Entries are sorted by priority (manual > contacts > CSV > auto-learned)
 * and by length (longer patterns match first to prevent partial matches).
 *
 * Storage: Room database with indexed spoken_form column for fast lookup.
 */
data class DictionaryEntry(
    val spokenForm: String,        // What the user says (lowercase)
    val writtenForm: String,       // What gets inserted
    val source: EntrySource,       // MANUAL, CONTACTS, CSV, AUTO_LEARNED
    val confirmations: Int = 0,   // For auto-learned: times user confirmed this correction
    val lastUsed: Long = 0L,      // Epoch millis — for LRU eviction
)

enum class EntrySource(
    val priority: Int,
) {
    MANUAL(100),
    CONTACTS(80),
    CSV(60),
    AUTO_LEARNED(40),
}

class PersonalDictionary(
    private val entries: List<DictionaryEntry>,
) {
    // Pre-sorted: highest priority first, then longest spoken form first
    private val sortedEntries = entries
        .filter { it.source != EntrySource.AUTO_LEARNED || it.confirmations >= 3 }
        .sortedWith(
            compareByDescending<DictionaryEntry> { it.source.priority }
                .thenByDescending { it.spokenForm.length }
        )

    private val patterns: List<Pair<Regex, String>> = sortedEntries.map { entry ->
        Regex("\\b${Regex.escape(entry.spokenForm)}\\b", RegexOption.IGNORE_CASE) to entry.writtenForm
    }

    fun apply(text: String): String {
        var result = text
        for ((pattern, replacement) in patterns) {
            result = pattern.replace(result, replacement)
        }
        return result
    }
}

Auto-Learning from User Corrections

Aawaaz pioneered auto-learning: when a user manually corrects a transcription (e.g., replaces "kubernetes" with "Kubernetes"), the system records the correction. After the same correction is observed N times (default: 3), it becomes a dictionary entry. This creates a feedback loop where the system improves with use. 

Kotlin/**
 * Tracks user corrections to auto-learn dictionary entries.
 * Called by the IME when the user edits committed text.
 *
 * Algorithm:
 *   1. Detect text replacement (composing region changed)
 *   2. If old text was from ASR and new text is user-typed:
 *      record (old → new) as a correction candidate
 *   3. When same (old → new) seen N times → promote to dictionary
 */
class CorrectionTracker(
    private val dao: DictionaryDao,
    private val threshold: Int = 3,
) {
    suspend fun recordCorrection(original: String, corrected: String) {
        if (original.equals(corrected, ignoreCase = true)) return
        if (original.isBlank() || corrected.isBlank()) return

        val existing = dao.findBySpokenForm(original.lowercase())
        if (existing != null) {
            dao.incrementConfirmations(existing.id)
        } else {
            dao.insert(
                DictionaryEntry(
                    spokenForm = original.lowercase(),
                    writtenForm = corrected,
                    source = EntrySource.AUTO_LEARNED,
                    confirmations = 1,
                    lastUsed = System.currentTimeMillis(),
                )
            )
        }
    }
}

Contacts Import (Android)

Aawaaz imports device contacts as dictionary entries, solving the #1 pain point in dictation: people's names. On Android, this requires READ_CONTACTS permission and a one-time sync. 

Kotlin/**
 * Import device contacts as personal dictionary entries.
 * Each contact's display name becomes a (spoken → written) mapping.
 *
 * Security: Only reads display names — no phone numbers or email addresses.
 */
suspend fun importContacts(
    contentResolver: ContentResolver,
    dao: DictionaryDao,
) = withContext(Dispatchers.IO) {
    val cursor = contentResolver.query(
        ContactsContract.Contacts.CONTENT_URI,
        arrayOf(ContactsContract.Contacts.DISPLAY_NAME_PRIMARY),
        null, null, null,
    ) ?: return@withContext

    cursor.use {
        val nameIdx = it.getColumnIndexOrThrow(
            ContactsContract.Contacts.DISPLAY_NAME_PRIMARY
        )
        while (it.moveToNext()) {
            val name = it.getString(nameIdx) ?: continue
            if (name.length < 2) continue
            dao.insertIfAbsent(
                DictionaryEntry(
                    spokenForm = name.lowercase(),
                    writtenForm = name,
                    source = EntrySource.CONTACTS,
                )
            )
        }
    }
}

Architecture

Kotlin/**
 * Voice Command Mode engine for the IME.
 *
 * Flow:
 *   1. User activates command mode (long-press or trigger word)
 *   2. ASR captures the spoken instruction
 *   3. Engine determines context: is text selected?
 *   4. If selected: send (selectedText + instruction) to LLM for transformation
 *   5. If not selected + search intent: open browser search
 *   6. If not selected + generation intent: generate text at cursor
 *   7. Replace selected text with LLM output OR insert at cursor
 *
 * Wispr Flow limits: 1–1000 words for selected text transformation.
 */
sealed interface CommandResult {
    data class TextTransformation(
        val original: String,
        val transformed: String,
    ) : CommandResult

    data class WebSearch(
        val query: String,
    ) : CommandResult

    data class TextGeneration(
        val generated: String,
    ) : CommandResult

    data class KeyAction(
        val keyCode: Int,
    ) : CommandResult
}

class VoiceCommandEngine(
    private val llm: Qwen3DictationFormatter,
) {
    private val searchPattern = Regex(
        """^(?:search\s+(?:for\s+)?|look\s+up\s+|google\s+)(.+)$""",
        RegexOption.IGNORE_CASE,
    )
    private val enterPattern = Regex(
        """^press\s+enter$""",
        RegexOption.IGNORE_CASE,
    )

    suspend fun execute(
        spokenCommand: String,
        selectedText: String?,
    ): CommandResult {
        // Check for key actions first
        if (enterPattern.matches(spokenCommand)) {
            return CommandResult.KeyAction(android.view.KeyEvent.KEYCODE_ENTER)
        }

        // Check for web search intent
        val searchMatch = searchPattern.find(spokenCommand)
        if (searchMatch != null && selectedText.isNullOrBlank()) {
            return CommandResult.WebSearch(searchMatch.groupValues[1])
        }

        // Text transformation or generation
        return if (!selectedText.isNullOrBlank()) {
            val prompt = """Transform the following text according to the instruction.
Instruction: $spokenCommand
Text: $selectedText
Return ONLY the transformed text."""
            val result = llm.format(prompt, foregroundPackage = null)
            CommandResult.TextTransformation(selectedText, result)
        } else {
            val prompt = """Generate text based on this instruction: $spokenCommand
Return ONLY the generated text."""
            val result = llm.format(prompt, foregroundPackage = null)
            CommandResult.TextGeneration(result)
        }
    }
}
